### Why two recipes

The two paths produce AURALLY similar audio (same voice, spinner-feel verbs) but through
completely different code. Discriminating them empirically is the only way to know which fix
to apply. Ben's quoted "Tinkering" could be either tense; a careful listen on real audio is
required.

## Reviewer findings — [tt1 · 2026-04-24T21:50:00+01:00]

### Code-path map — two distinct paths produce spinner-verb-style audio

**Path A — heartbeat verb (present-continuous):**

1. Renderer tick fires at POLL_INTERVAL_MS, reads `workingSessionsCache`, calls
   `decideHeartbeatAction` (`app/lib/heartbeat.js:120-153`).
2. If conditions met → picks verb via `pickHeartbeatVerb` (`heartbeat.js:89-95`). Returns
   either a `SPINNER_VERBS` entry (60%) or a `THINKING_PHRASES` entry (40%). SPINNER_VERBS
   includes `'Tinkering'`, `'Moonwalking'`, `'Flibbertigibbeting'`, etc.
3. Renderer calls `window.api.speakHeartbeat(verb, sessionShort)` (`renderer.js:299-300`).
4. IPC handler `speak-heartbeat` (`app/lib/ipc-handlers.js:584-626`):
   - Validates verb `/^[A-Za-z][A-Za-z ]{1,59}$/`
   - Validates sessionShort `/^[a-f0-9]{8}$/`
   - Writes filename `${ts}-H-0001-${sessionShort}.mp3` (line 614)
   - Calls `callEdgeTTS(verb, voice, outPath)` — ALWAYS edge-tts, NEVER OpenAI (no provider
     branch in this handler — heartbeat is free-TTS-only by design)
5. Clip appears in `~/.terminal-talk/queue/` with H-prefix.
6. Queue-watcher picks it up, adds to renderer's `queue` array.
7. `playNextPending` picks next clip by priority/focus/mtime; heartbeat clips compete with body
   clips as plain queue entries.
8. `audioPlayer.playPath` reads filename, runs `isHeartbeatClip(fn)` → true → volume
   `0.45 × masterVolume` (`audio-player.js:278-288`).
9. After play-end, `onClipEnded` + `scheduleAutoDelete` tear down the clip (ephemeral, quick
   auto-delete; not subject to the body auto_prune_sec delay).

**Path B — end-of-turn footer verb (past-tense):**

1. Stop hook fires. `hooks/speak-response.ps1` computes `elapsedSec` and attempts UIA scrape for
   the terminal's own footer text.
2. If scrape succeeds → uses scraped phrase. If fails → `synth_turn.py` falls back to
   `format_elapsed_phrase(elapsed_sec)` (`synth_turn.py:1214-1244`).
3. `format_elapsed_phrase` picks a random verb from **`PAST_TENSE_VERBS`** (`synth_turn.py:
   1194-1211`) — a tuple containing `'Tinkered'`, `'Moonwalked'`, `'Sautéed'`, `'Flibbertigibbeted'`,
   `'Pondered'`, etc. Formats as `"<Verb> for N second(s)"` or `"<Verb> for M minute(s) and N
   second(s)"`.
4. This string gets **APPENDED as a body clip** — no H-prefix, no T-prefix, full body volume,
   subject to auto_prune_sec like any other body clip.
5. Plays as the final clip of the turn at 1.0 × master volume.

### Most likely scenario to check first

Given Ben's specific word "**Tinkering**", H1 is the first to test because:

- `PAST_TENSE_VERBS` at `synth_turn.py:1208` contains `'Tinkered'`. Spoken by edge-tts, at
  normal UK voice cadence, the -ed suffix becomes -t or -id depending on the voice model.
  "Tinkered" heard quickly can phonetically register as "Tinkering" to the listener. "Sautéed"
  lands as "so-tayed" which sounds verbing-like too.
- It's a BODY clip by design, plays at full volume, sits on the dot strip until auto-prune —
  matches Ben's description "leaking into real transcript" more closely than the quiet
  heartbeat path.
- There's a Ben-specific comment at `synth_turn.py:1190-1191`: *"Sautéed explicitly included
  because the terminal uses it and it's what Ben pointed at in the transcript."* Indicates
  this clip has previously confused Ben — the V3 list was CHOSEN to mirror Claude Code's own
  verb set, but the aural effect is the same verb-style as heartbeats.

**If TT2's empirical test confirms H1:** fix is design-level. Options to put in front of Ben:
  1. Opt-out setting (`playback.speak_elapsed_phrase: false`) — hide the footer entirely for
     users who find it intrusive.
  2. Voice contrast — synthesise the footer clip with a different edge-tts voice so it reads
     as a distinct "meta" layer (like `en-GB-LibbyNeural` vs `en-GB-RyanNeural` for the body).
  3. Volume reduction — tag the footer clip with a new prefix (`-F-` for footer?) and add a
     classifier rule so it plays at, say, 0.6 × master.

**If H2 is confirmed instead (or additionally):** fix is a gate in `decideHeartbeatAction` —
when `workingSessionsCache` is non-empty AND a body-clip for that session has landed within
the last N seconds, skip heartbeat emission. Or simpler: track a per-session `firstBodyClipAt`
and forbid heartbeat until N seconds *after* the turn's first body clip has played out (letting
heartbeat fire ONLY in the pre-first-audio window, not during gaps between body clips).

### Pattern note (adjacent risk, not bug)

Both V1 and V3 lists are maintained separately and can drift. Adding a verb to `SPINNER_VERBS`
but not `PAST_TENSE_VERBS` (or vice-versa) doesn't break functionality but creates aural
inconsistency. **Out of scope for #2** but worth flagging for a future QUEUE item: "unify
spinner-verb lexicon into a single source with morphology function".

## Causality — [tt2 · 2026-04-25T02:20 — CODE-INVESTIGATION RESULT]

Traced both TT1-proposed paths plus the Path C (dictation overlap) scenario Ben originally
experienced. Conclusion: **#2 should close as fixed-by-Path-C-mic-pause-work + Path A is a
narrow race + Path B is by-design footer (opt-out candidate).**

### Path A — heartbeat clip leaks during body playback

**Code trace:**
- `app/lib/heartbeat.js:147` — `if (isQueueActive) return { type: 'reset-silent', … };`
- `app/renderer.js:160-169` — `isQueueActive()` returns true if audio is playing OR there
  are fresh-unheard-unmuted queue items within the last 60 s.

So: once a body clip lands in the queue (isQueueActive=true), decideHeartbeatAction resets
the silent timer. Heartbeat won't emit again until 5 s of continuous silence.

**Race window:** heartbeat tick fires between (a) user pressing Enter and (b) body clip
arriving. If the user's last-seen clip was >60 s ago (isQueueActive=false) AND silence >
initialMs (5 s) AND last-heartbeat > intervalMs (8 s), `decideHeartbeatAction` emits.

This produces ONE heartbeat clip (at 45% volume, H-prefix) that plays just BEFORE the body
clip. NOT during body playback — by that point, isQueueActive=true and heartbeat is gated.

**Severity:** low. One quiet "Tinkering" before body audio arrives is tolerable UX. If Ben
observed it as disruptive, the fix is either:
- Extend `ACTIVE_FRESH_MS` gating to "any queue activity within last 10 s" (closes race)
- Gate heartbeat on "time since last user-input" (needs new IPC to track Enter-press)

### Path B — end-of-turn footer "Tinkered for X seconds"

**Code trace:**
- `app/synth_turn.py:1186-1244` — `PAST_TENSE_VERBS` + `format_elapsed_phrase` build a
  body-text clip like `"Tinkered for 45 seconds"` appended at Stop hook.
- Plays at full volume as body audio. By design (mirrors Claude Code's terminal spinner
  "Cooked for 49s" footer).

**This is the design, not a bug.** If Ben wants to turn it off, that's a feature request
→ new QUEUE item: "add `playback.footer_enabled` toggle". Not #2's scope.

### Path C — heartbeat during dictation (Wispr/mic-captured state)

**This was the actual bug Ben reported.** Fix shipped 2026-04-23 via HB4 split into
`_micCaptured` + `_systemAutoPaused` two-flag model. Verified clean in Surface D audit
(#17).

**Ben confirmed** during this session: *"Path C was a problem I had but then we fixed
that. Now everything's paused while I'm dictating, which is great."*

### "Tinkering" vs "Tinkered" — present vs past

Ben's original report used "Tinkering" (present continuous). But:
- **Present-continuous verbs** (`Tinkering`, `Moonwalking`) only come from
  `heartbeat.js SPINNER_VERBS` on the JS side, played as low-volume H-prefix clips.
- **Past-tense verbs** (`Tinkered`, `Moonwalked`) only come from
  `synth_turn.py PAST_TENSE_VERBS` in the end-of-turn footer, played at full volume.

Python never synthesises present-continuous verbs. JS never synthesises past-tense verbs.
The two systems are cleanly separated. So:
- "Tinkering at full volume during body" is IMPOSSIBLE via any normal code path.
- "Tinkering at low volume between turns" is expected heartbeat behaviour.
- "Tinkered for Xs at end of turn" is the by-design footer.

### Final diagnosis

- **Root cause (of original #2 report):** Path C mic-pause. Fixed. Verified.
- **Is there anything else observably broken?** No. Path A is a narrow race with low-

- **Smallest fix:** none required. Close #2. Open a follow-up (optional) to
