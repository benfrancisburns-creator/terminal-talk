## 0. Ground rules

### Regression testing cadence — applies to every item below

**Before starting any item (lockdown):**

```
npm run lint              # 0 errors expected
npm run knip              # 0 findings expected
python -m ruff check app/*.py   # "All checks passed" expected
npm run check:file-length # OK expected
node scripts/run-tests.cjs --logic-only   # 195/195 expected
```

Any non-zero exit means **STOP** — something was broken before the work began. Fix first, then proceed.

**After each discrete change within an item:**

Re-run the SAME lockdown subset. If a test that was green now fails, that's a regression introduced by the change in hand. Investigate before moving on — don't batch.

**At item finish (acceptance):**

- Full test harness: `npm test` (268/268 expected, grows as new tests land).
- Full lint chain + knip + ruff + file-length + doc-drift all 0.
- If item touches renderer or main.js: `npx playwright test` (25/25 expected).
- If item touches `app/` code: `node scripts/sync-app-mirror.cjs` to keep the kit mirror in step; kit demo loads without errors.
- Commit with finding-ID + test count delta in the body.

### "Whatever we find we fix" rule

Items with surprises surfaced during execution:
- If the surprise is small (≤ 15 min) and in-scope, fix in the same commit.
- If large or out-of-scope, **stop and append to this doc** as a new EX-N item. Don't improvise off-plan.

## 2. EX1 — Absolute-path spawn for `taskkill` + `powershell`

### What

S4 Sonar flagged 6× `javascript:S4036` "PATH variable may be hijacked" hotspots in `app/main.js` + `scripts/verify-voices.cjs`. Triage accepted 4 of them under the same-user-trust-boundary (same argument as D2-4 IPC integrity), but explicitly filed this follow-up: **`taskkill` and `powershell` live in `C:\Windows\System32\` on every Windows install and can safely use absolute paths. Zero compat risk.**

`python` is trickier (installed in user-space, varies by install type) and stays deferred to a separate session.

### Files touched

- `app/main.js` — 3 call sites:
  - `main.js:1562` `execFileSync('powershell', ...)` (orphan cleanup)
  - `main.js:1578` `spawn('taskkill', ...)` (process termination)
  - (also in S4 hotspot list: there may be a second powershell call — verify before editing)

### Plan

1. **Lockdown** (see §0).
2. Define a constant at top of main.js: `const SYSTEM32 = process.env.SystemRoot ? path.join(process.env.SystemRoot, 'System32') : 'C:\\Windows\\System32';`. Using `SystemRoot` env makes it portable across weird installs (corporate images sometimes relocate Windows dir).
3. Replace `spawn('taskkill', ...)` → `spawn(path.join(SYSTEM32, 'taskkill.exe'), ...)`.
4. Replace `execFileSync('powershell', ...)` → `execFileSync(path.join(SYSTEM32, 'WindowsPowerShell\\v1.0\\powershell.exe'), ...)` (the actual install path; not just `powershell.exe` in System32 — PS 5.x lives in a versioned subfolder).
5. Verify the SonarQube resolved-safe comments still apply (they do — this is strengthening, not changing the rationale).
6. **Regression after each swap:** run logic-only tests. Also run `npx playwright test tests/e2e/launch.spec.ts` — that exercises the IPC + orphan-cleanup path if the tests close the window.

### Acceptance

- `grep -n "spawn('taskkill'" app/main.js` returns nothing.
- `grep -n "execFileSync('powershell'" app/main.js` returns nothing (or only in quoted strings, not executable code).
- All 268 logic + 25 E2E tests still pass.
- `npm run sonar` after: all 6 hotspots still marked reviewed-safe; no new findings.

### Commit shape

`fix(security): absolute paths for taskkill + powershell spawns (S4 EX1)` with body citing the S4 hotspot IDs.

## 4. EX3 — Reload-renderer menu item (product)

### What

Occasionally the toolbar gets into a weird visual state (stale assignments, stuck overlay, post-devicechange orphan). Today the fix is quit + restart Electron. Adding a keyboard shortcut (or tray-menu item, or settings-panel button) that just **reloads the renderer** — not the whole process — gives the user a cheap recovery path.

### Design choice

Three surfaces possible; pick one:

**Option A — Keyboard shortcut** (`Control+Shift+R` global, or `Ctrl+R` window-scoped).
  - Pro: fastest. Pro: discoverable via shortcuts table.
  - Con: conflicts with user's own shortcuts potentially. Window-scoped (not global) is safer.

**Option B — Settings panel button** (`Reload toolbar` under "Playback" or in a new "Troubleshooting" section).
  - Pro: discoverable. Pro: no shortcut conflict.
  - Con: requires opening the panel first.

**Option C — Right-click on the bar → context menu with "Reload"**.
  - Pro: zero surface added elsewhere.
  - Con: hidden feature.

**My recommendation:** A + B together. Ctrl+R window-scoped (browser convention) + a Settings-panel button with the keyboard hint. Belt + braces.

### Files touched (Option A+B)

- `app/main.js` — register `webContents.on('before-input-event')` or equivalent for the Ctrl+R binding; call `win.webContents.reloadIgnoringCache()`.
- `app/index.html` — add button in the Playback section (or a new Troubleshooting section) with `id="reloadToolbar"`.
- `app/renderer.js` — wire button click to an IPC call.
- `app/preload.js` — expose `reloadToolbar` on `window.api`.
- `tests/e2e/settings.spec.ts` — add test for button existence + click (mocking the reload).

### Plan

1. **Lockdown.**
2. Design-first: confirm with Ben that A+B is what he wants. He mentioned this thread at 2026-04-20 with "my vote: kit reset button now (tiny) + renderer-only reload as a product menu item (also small) later. The destructive stuff needs more thought around confirm dialogs and won't ship clean in a single patch."
3. Implement IPC first (`main.js` handler + preload expose).
4. Add button to settings panel HTML.
5. Wire click handler in renderer.js to call the IPC.
6. Register Ctrl+R binding via `webContents.on('before-input-event')`.
7. Add E2E test that asserts the button exists and is clickable.
8. **Regression:** after each step, lockdown. End-of-item: full E2E including the new test.

### Acceptance

- Ctrl+R inside the toolbar window reloads the renderer (dots repaint, panel closes).
- Settings-panel button with a label like "Reload toolbar" exists and triggers the same action.
- E2E test covers the button.

### Commit shape

`feat(ux): renderer-reload menu + Ctrl+R keyboard shortcut (EX3)`.

## 5. EX4 — Undo-clear for deleted clips

### What

The trash icon in the toolbar (`#clearPlayed`) deletes played clips permanently. No undo. If the user accidentally clears clips they wanted to keep, they're gone.

### Design choice

Needs Ben's pick between:

**Option A — Time-windowed undo.** After clear, show a transient toast ("3 clips cleared — Undo") for 10 s. Click undo restores the clips into the queue.
  - Pro: browser convention; low learning curve.
  - Con: requires a toast component (new UI primitive).

**Option B — Confirmation dialog.** Before clear, show "Clear 3 played clips? [Cancel] [Clear]".
  - Pro: simplest to build (existing dialog primitive).
  - Con: friction on every clear, not just the accidental ones.

**Option C — Soft delete + restore menu.** Deleted clips move to a hidden `clips/.trash/` dir; a "Recently cleared" submenu in Settings shows them for N hours.
  - Pro: most forgiving.
  - Con: two-step UX; users forget the restore panel exists.

**My recommendation:** Option A. Fits the "transient, cheap, forgiving" mental model most apps have trained users for.

### Not starting until Ben picks

This one opens a design conversation. Write a dialogue summary + `DIALOGUE-PROMPT.md` stub, and **wait on Ben** before implementing.

## 7. EX6 — `app/main.js` refactor

### Why

1,802 lines. Fails "cognitive-load" threshold (Robert C. Martin's 200-line ideal; pragmatic 500-line cap). Holds five distinct concerns that should live in their own modules:

1. Window lifecycle + dock snapping (~250 lines)
2. IPC handlers (~600 lines — the biggest chunk)
3. Palette registry + assignments (~300 lines — already partially extracted to `app/lib/palette-alloc.js` + `session-stale.js`)
4. Process watchdog + orphan sweep (~200 lines)
5. Config + safeStorage (~200 lines — partially in `app/lib/api-key-store.js` + `config-validate.js`)

### Target end-state

```
app/main.js                 ~800 lines  (orchestration only)
app/lib/window-dock.js      new ~250
app/lib/ipc-handlers.js     new ~600
app/lib/queue-watcher.js    new ~200
app/lib/watchdog.js         new ~200
app/lib/config-store.js     new ~200  (wraps loadConfig / saveConfig with validator + safeStorage)
```

**`app/main.js` becomes:** requires + constants + `app.whenReady()` sequence + `app.on(...)` lifecycle + top-level `BrowserWindow` creation. The heavy stuff lives in `lib/`.

### Why it's high-risk

Every module extraction is a rename + import restructure that touches dozens of internal references. A mis-wired IPC handler could silently break a specific interaction that only E2E catches. **This needs its own focused session.**

### Plan (to be expanded to its own ULTRAPLAN-MAIN.md before execution)

1. **Snapshot coverage + test counts** — record exact baseline numbers before touching anything.
2. Extract ONE module at a time, in dependency order:
   a. `config-store.js` first (no downstream deps)
   b. `window-dock.js`
   c. `queue-watcher.js`
   d. `watchdog.js`
   e. `ipc-handlers.js` last (imports from all of above)
3. After each extract:
   - Full `npm test` (logic-only).
   - Full `npx playwright test`.
   - `npm run lint` + knip + ruff + file-length.
   - `npm run sonar` if the extracted module is > 100 lines — lets us see the complexity score decrease in real time.
4. Update `file-length-baseline.json` with each commit.
5. Do NOT rebase-squash into one commit — one extract per commit means git-bisect can find the specific extraction that broke something.
6. Ratchet `file-length-baseline.json`'s ceiling from 3000 → 2000 in the final commit.

### Acceptance

- `app/main.js` under 1000 lines (stretch: under 800).
- All 268 logic + 25 E2E still green.
- Coverage floor held or improved (EX6 gains unit-testable lib/* modules → higher coverage).
- No new Sonar findings.

### Defer-anchor condition

If any single module extraction takes > 90 minutes or breaks >3 tests, **stop the session**, commit what's extracted cleanly, park the rest. The refactor doesn't have to finish in one sitting.

## 10. EX9 — Extend doc-drift checker to scan code comments

### Why

v0.3.0 assessment (§5.4) flagged N4: `app/session-registry.psm1:91-93` docstring misstated main.js's auto-prune policy. N4 was a **code comment** contradicting actual **code behaviour**. Our `scripts/check-doc-drift.cjs` scans docs prose but not code comments, so it didn't catch it.

Assessment suggested extending the drift checker with assertion patterns like:

```js
{
  pattern: /main\.js['s ]*(no[- ]auto[- ]prune|never prune)/i,
  description: 'main.js.ensureAssignmentsForFiles DOES auto-prune non-pinned sessions past 4h grace.',
  skip: [],
}
```

### Files touched

- `scripts/check-doc-drift.cjs` — extend the file-type filter to include `.psm1`, `.ps1`, `.py`, `.js` (currently just `.md/.html/.js/.css/.txt`). Add a new rule category `code-comment-invariant` with stricter pattern matching.
- `ASSESSMENTS/S7-deps-functional-filelen/findings.md` — update to reflect it's no longer a gap.
- Add ~5 concrete rules targeting the patterns most likely to drift: auto-prune policy, grace-window duration, palette size, etc.

### Plan

1. **Lockdown.**
2. Read the existing check-doc-drift.cjs structure (125 lines, so small).
3. Extend file-type matcher.
4. Add a new `CODE_COMMENT_RULES` array alongside the existing `DOC_RULES`.
5. Add 5 rules: auto-prune (catches N4), palette-size (24 not 32), grace-window (4h not 2h), clips-dir (queue/ not clips/), tokens-source.
6. Run against the tree, should pass.
7. Simulate a regression: temporarily re-insert the old N4 wording in session-registry.psm1, verify the checker catches it.
8. Revert the simulated regression.

### Acceptance

- `npm run check-doc-drift` (or direct invoke) still exits 0 on main.
- Manual regression simulation fires the new rule.
- CI doc-drift job covers it.

### Commit shape

`feat(ci): code-comment invariants in doc-drift checker (EX9)`.
