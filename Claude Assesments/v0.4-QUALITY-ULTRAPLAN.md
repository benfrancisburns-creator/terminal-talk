# v0.4 Quality tier — master plan

**Date:** 2026-04-20
**Scope:** rigorous traditional QA across the terminal-talk codebase. Static analysis (Knip, ESLint, ruff, PSScriptAnalyzer, SonarCloud) + testing pyramid expansion (unit, functional, E2E) + dependency audit + coverage floor + file-length gate.
**Policy:** **fix everything we find** unless there is an explicit, documented reason not to. No "too hard", no "low priority" — reasons must be mechanical and concrete.

---

## 1. Phase model (why parallelism works)

Running 5 terminals against 5 tools at once sounds chaotic, but it works cleanly if we separate **discovery** from **fix**:

| Phase | Parallel? | Safe because |
|---|---|---|
| **1. Discovery** | Yes — 5 terminals | All tools are read-only. They write findings to `ASSESSMENTS/<stream>/findings.md`, a file per stream. Zero source-code conflicts. |
| **2. Triage** | No — single lane + Ben review | Classify each finding: fix / defer-with-rationale / accept. Merge into one dispositions list. |
| **3. Fix** | Carefully partitioned | Language-based lanes per `feedback-multi-terminal-coord.md`. JS fixes serialized if two streams both edit JS (Knip before ESLint is the natural order). |
| **4. Lock-in** | Sequential | One tool at a time wired into CI as blocking. Never promote a tool to blocking before its findings are all dispositioned. |
| **5. Ongoing** | CI-driven | Future commits fail if they regress any of the new checks. |

**Ship cadence:** one patch release per tool lock-in (v0.4.1 Knip, v0.4.2 ESLint, …). Or bundle all of Phase 4 as a single v0.4.0 "quality tier" tag. My preference: bundle — one tag "quality groundwork complete" reads cleaner in history than 8 patch bumps.

## 2. Finding protocol — one format for every tool

Every finding, from any tool, uses this schema. Lives in the stream's `findings.md`. One block per finding.

```markdown
### F<NN> — <one-line title in imperative tense>
- **File:** `path/to/file.js:123` (or `:123-130` for a range)
- **Tool:** Knip / ESLint / ruff / PSScriptAnalyzer / SonarCloud / c8 / jscpd / manual
- **Severity:** blocker | critical | major | minor | info
- **Evidence:**
  ```
  <verbatim tool output OR quoted source excerpt>
  ```
- **Why it matters:** 1-3 sentences. What breaks, what risks regressing, what it slows down. No hedging.
- **Proposed fix:** concrete action. "Delete function X. Inline its single caller into Y." Not "consider refactoring".
- **Effort estimate:** ~N min
- **Disposition:** [ ] fix | [ ] defer (reason: …) | [ ] accept (reason: …)
- **Landed in commit:** `<sha>` (filled on fix; blank while open)
```

Severity rubric:
- **blocker** — user-facing bug or security issue. Ship date slips until fixed.
- **critical** — likely to cause a production bug within 90 days. Fix in this tier.
- **major** — code smell or quality issue with real cost. Fix unless a mechanical reason blocks it.
- **minor** — stylistic or hygiene. Fix unless friction > benefit.
- **info** — observations. Can be noted and moved on.

### S1 — Knip (JS unused code + exports + dependencies)

**Purpose.** Find and delete unused files, unused exports across `app/*.js`, `app/lib/*.js`, `scripts/*.cjs`, and the kit's `docs/ui-kit/*.js`. Also flag unused `package.json` dependencies.

**Why first.** Dead code extraction should precede any other JS tool. ESLint will have less to look at; SonarCloud will grade a cleaner codebase.

**Setup (~15 min).**
1. `npm install --save-dev knip`
2. Write `knip.json` declaring entry points:
   - `app/main.js` (Electron main process entry)
   - `app/preload.js` (Electron preload)
   - `app/renderer.js` (loaded by app/index.html)
   - `scripts/run-tests.cjs` (test harness)
   - `docs/ui-kit/kit-bootstrap.js` (kit demo entry)
   - Every `scripts/*.cjs` that's referenced by an npm script
3. Add `ignoreDependencies` if any dev-only deps are picked up by stealth (electron itself, c8)
4. Add `npm run knip` script → `node_modules/.bin/knip`
5. Add `npm run knip:fix` → `knip --fix` (dry-run first)

**Baseline command.** `npm run knip > ASSESSMENTS/S1-knip/baseline.txt`

**Expected findings on our codebase.**
- Likely unused helpers in `app/main.js` and `app/renderer.js` accumulated during refactors (particularly around D2-3 kit iframing — some old kit-related exports might be orphaned)
- Possibly unused entries in `app/lib/voices-window.js` (generated file, might have stale entries)
- Unused `devDependencies` are possible but we keep it lean

**Known pitfalls.**
- Kit files loaded at runtime via `fetch` (kit-bootstrap.js loads app-mirror/ files) — declare `docs/app-mirror/index.html` + `docs/app-mirror/renderer.js` as entries so they aren't flagged unused
- Generated files (`tokens-window.js`, `palette-classes.css`, `tokens.mjs`) are consumed via `<script>` / `<link>` tags not import — mark as entries or `ignore` patterns
- PS / Python files are outside Knip's scope — don't try to force coverage there

**Acceptance criteria.**
- `npm run knip` returns zero findings OR every remaining finding is dispositioned as accept/defer with reason
- CI step added as advisory first, then blocking

**Lock-in (CI).** Add `knip` step to `.github/workflows/test.yml` under `doc-drift` job.

### S2 — ESLint (JS style + common bug patterns)

**Purpose.** Catch unused vars, unreachable code, accidental globals, shadowed variables, inconsistent style. Enforce a minimal standard across the ~4,000 lines of JS.

**Setup (~30 min).**
1. `npm install --save-dev eslint @eslint/js globals` (flat config, modern ESLint)
2. Write `eslint.config.js` with:
   - `globals.browser` for renderer.js (document, window, fetch)
   - `globals.node` for main.js, scripts/
   - `globals.electron` custom for preload API if needed
   - Rule set: `@eslint/js recommended` + a handful of custom rules:
     - `no-unused-vars` with `argsIgnorePattern: '^_'`
     - `no-console: off` (we use diag() intentionally)
     - `eqeqeq: error`
     - `no-var: error`
     - `prefer-const: warn`
     - `no-implicit-globals: error`
     - `no-param-reassign: warn`
3. Add `npm run lint` → `eslint app/ scripts/ docs/ui-kit/`

**Baseline.** `npm run lint > ASSESSMENTS/S2-eslint/baseline.txt 2>&1`

**Expected findings.**
- Some `let` that could be `const` in main.js/renderer.js
- A handful of shadowed variables in nested scopes
- Maybe the odd `==` that should be `===`
- Possible unused args in callbacks

Historical codebases of this size with no prior linter typically produce 50–200 findings of which 80% are trivially auto-fixable.

**Known pitfalls.**
- Electron preload's `contextBridge` API confuses older ESLint configs — verify the preload rules don't flag the pattern
- `docs/ui-kit/mock-ipc.js` uses browser globals + a lot of IIFE patterns — may need per-file overrides
- Build-generated files MUST be in `ignores`: `app/lib/tokens-window.js`, `app/lib/palette-classes.css` (not lintable anyway), `docs/ui-kit/tokens.mjs`, `docs/app-mirror/**`

**Acceptance criteria.** Zero errors. Warnings tolerated but each dispositioned.

**Lock-in.** `eslint --max-warnings=0` in CI.

### S3 — Python + PowerShell lint

**Purpose.** Parity for non-JS code. `ruff` handles Python (fast, thorough, replaces flake8+isort+pylint for most purposes). `PSScriptAnalyzer` handles PowerShell.

**Python setup (~15 min).**
1. `pip install ruff` (or add to a dev-requirements.txt if we grow one)
2. Create `ruff.toml` at repo root:
   ```
   line-length = 100
   target-version = "py38"
   select = ["E", "F", "W", "I", "B", "C90", "UP"]
   ignore = ["E501"]  # line length handled separately; synth_turn has long strings
   ```
3. Exclusions: none needed (we only have 4 Python files)
4. Add `npm run lint:py` → `ruff check app/*.py`

**Expected findings.**
- Some unused imports in `synth_turn.py`
- Possibly complex-function warnings (synth_turn.run is long)
- Maybe shadowed builtins

**PowerShell setup (~15 min).**
1. Install PSScriptAnalyzer (already on Windows Powershell 5+; install via `Install-Module -Name PSScriptAnalyzer` if missing)
2. Write `PSScriptAnalyzerSettings.psd1` with:
   - Include default rules
   - Exclude `PSUseDeclaredVarsMoreThanAssignments` (noisy on PS modules)
   - Severity: Warning and above
3. Add `scripts/lint-ps.ps1` → invokes `Invoke-ScriptAnalyzer -Path hooks/,app/*.ps1,app/*.psm1,install.ps1,uninstall.ps1 -Settings PSScriptAnalyzerSettings.psd1`

**Expected findings.**
- Some singular/plural noun-verb style issues
- Possibly unapproved verbs (`Touch-Or-Assign-Session` memory note from assessment)
- Global variable misuse

**Known pitfalls.**
- PSScriptAnalyzer on GitHub Actions Ubuntu runners requires `pwsh` setup — use `microsoft/ps-analyzer-action` or install via `dotnet install`
- Some hook scripts embed inline strings that trigger false positives — may need `# PSScriptAnalyzer disable` pragmas

**Acceptance criteria.** Zero ruff errors. PSScriptAnalyzer zero Warning-or-above findings (Info tolerated).

**Lock-in.** Separate CI jobs: `python-lint` (runs ruff), `powershell-lint` (runs PSScriptAnalyzer).

### S4 — SonarQube self-hosted (multi-language complexity + smells + duplication)

**Purpose.** Industry-standard code-quality grading. Cyclomatic complexity per function, maintainability grade, duplication %, technical-debt estimate, coverage integration. Gives a before/after number for the big-file refactor work that comes later.

**Setup (~45 min, most of it Ben's server + scanner install).**

Ben has SonarQube downloaded locally. Self-hosted means the scans run on his laptop and push to his localhost:9000 server — we don't integrate with GitHub Actions CI (the runners can't reach his laptop). Trade-off: no automated PR blocking, but full analysis capability. Fine for a solo dev.

1. **Ben action:** Start SonarQube. Depending on distribution:
   - ZIP: `bin\windows-x86-64\StartSonar.bat`
   - Docker: `docker run -d -p 9000:9000 --name sonar sonarqube:community`
   - MSI/installer: run from Start menu, usually a service
2. **Ben action:** Open [http://localhost:9000](http://localhost:9000). Default admin/admin, prompted to change.
3. **Ben action:** Create project → name `terminal-talk`, key `terminal-talk`. Pick "Locally" as analysis method. Generate a user token — copy it somewhere safe (one-time reveal).
4. **Ben action:** Install `sonar-scanner` CLI. Download from sonarsource.com/sonar-scanner, unzip somewhere, add `bin/` to PATH. Verify with `sonar-scanner --version`.
5. **I can do:** Write `sonar-project.properties` at repo root:
   ```
   sonar.projectKey=terminal-talk
   sonar.host.url=http://localhost:9000
   sonar.sources=app,scripts,hooks
   sonar.exclusions=**/node_modules/**,**/tests/baselines/**,docs/app-mirror/**,app/lib/tokens-window.js,app/lib/palette-classes.css,docs/ui-kit/tokens.mjs
   sonar.tests=tests,scripts/run-tests.cjs
   sonar.javascript.lcov.reportPaths=coverage/lcov.info
   ```
   Token lives in `~/.sonarqube-token` (gitignored) or passed via `SONAR_TOKEN` env var — NOT committed.
6. **I can do:** Add `npm run sonar` → `sonar-scanner -Dsonar.token=$SONAR_TOKEN`

**Baseline workflow.**
1. Ben runs `npm run test:coverage` to generate `coverage/lcov.info`
2. Ben runs `npm run sonar`
3. Ben opens localhost:9000 → terminal-talk project → sees full dashboard
4. **I do:** export findings to `ASSESSMENTS/S4-sonar/baseline.md` via SonarQube's Web API (`/api/issues/search?componentKeys=terminal-talk&resolved=false`) — I can write a little script that fetches + formats them into our standard finding schema.

**Expected findings.** Plausible categories:
- Complexity: `main.js` and `renderer.js` have long functions that exceed the default 15 cyclomatic threshold (probably several each)
- Cognitive complexity: nested conditionals in `ensureAssignmentsForFiles` or `playNextPending`
- Duplication: some (we've aggressively deduped via CC-1..CC-8, but a 3–5 % duplicate rate is normal)
- Security hotspots: some `child_process.spawn` patterns might be flagged for review (most are safe — we sanitize argv)

**Known pitfalls.**
- SonarQube's JS analyzer is strict about "cognitive complexity" — some legitimately necessary functions (like ensureAssignmentsForFiles) may need `// NOSONAR` with rationale
- Don't disable rules globally; disable per-finding with justification comment
- The `tests/baselines/palette/` PNG dir must be excluded or Sonar tries to "analyze" them
- First scan can take several minutes on a 40 MB project; don't cancel it
- Community Edition has no PR decoration — findings only visible in the local UI

**Acceptance criteria.** Maintainability grade ≥ A. All blocker + critical findings addressed. Major findings triaged (fix or documented-defer).

**Lock-in (local, not CI).** Add a pre-commit hook or `npm run sonar` invocation as part of the release checklist. No PR-blocking possible with self-hosted on a single laptop; relies on discipline. Can be revisited if Ben decides to host SonarQube on a reachable server (AWS/Hetzner/VPS).

### S5 — Coverage baseline + unit-test gap fill

**Purpose.** c8 is already in CI but log-only. Characterise the current coverage floor; find untested modules; write unit tests for critical uncovered paths; set CI floor to "can only go up".

**Setup (~20 min).**
1. Run `npm run test:coverage` locally, capture lcov report
2. Parse per-file coverage. Expect `app/lib/*.js` to be well-covered (most exist because of the test harness). Expect `app/main.js` and `app/renderer.js` to be poorly covered (large + testing needs DOM/Electron harness).
3. List every module with < 70 % line coverage → unit-test backlog
4. Decide per module: is logic extractable to `app/lib/` (testable) or inherently integration-only (stays untested at unit level, relies on E2E)?

**What to write.** For each extractable-but-untested module:
- Pull the testable logic into `app/lib/<name>.js` if not already
- Add unit tests in `scripts/run-tests.cjs` under a new `describe` block
- Verify coverage climbs

**Expected gap list.** Plausible guesses before the actual run:
- `app/main.js:callEdgeTTS` timeout logic
- `app/main.js:ensureAssignmentsForFiles` integration with `allocatePaletteIndex`
- `app/renderer.js:playNextPending` full branch coverage (priority, focus, pending, fallback)
- `app/renderer.js:scheduleAutoDelete` with various toggle states
- `app/synth_turn.py:_run_edge_tts` retry + timeout paths

**Known pitfalls.**
- DOM-dependent renderer code can't be unit-tested without jsdom. Don't force it — flag for E2E instead.
- c8 doesn't cover `.py` or `.ps1`. Python coverage via `coverage.py`, PS via Pester if we add it (probably overkill).
- Setting a "floor that only goes up" requires storing the current pct somewhere CI can compare against (a tiny baseline file in `.github/`).

**Acceptance criteria.** Coverage floor set. Every `app/lib/*.js` module at 80%+ line coverage. Gap list triaged — each item either has a test or a reason-for-not.

**Lock-in.** c8 CI job promoted from log-only to failing if coverage drops below floor.

### S6 — E2E test coverage expansion (Playwright)

**Purpose.** We have 13 E2E tests. They cover launch, some sessions scenarios, and settings. Feature coverage is patchy. Identify user-facing flows without E2E coverage and add tests.

**Setup.** Already installed; no new infra.

**Baseline (~20 min).** Enumerate user-facing features from `README.md` + settings panel + the design-system pages. Cross-reference against `tests/e2e/*.spec.ts` names. Produce a gap list.

**Likely gaps (before enumeration):**
- Hey-jarvis highlight-to-speak (priority queue)
- Auto-continue-after-click new setting (v0.3.6)
- Session close → stale-session greying (v0.3.3 guard)
- Mute/unmute toggle effect on queue
- Settings panel auto-prune toggle + delay input
- Palette cycling (right-click session → cycle colour)
- The clear-all-played button
- Electron window restore-position across quit/relaunch
- Multi-monitor positioning (if testable)

**What to write.** One `*.spec.ts` per feature gap, following existing patterns in `tests/e2e/sessions.spec.ts`. Keep tests fast — each test ≤ 10s ideally.

**Expected scope.** 8–15 new E2E tests, bringing total to 21–28.

**Known pitfalls.**
- Audio playback in headless Electron is tricky — might need the silent-WAV shim pattern from `mock-ipc.js`
- Hey-jarvis tests need simulated global-shortcut triggers — Playwright can't do Windows-level shortcuts, may need IPC injection path
- Flakiness — any test taking > 5 s needs investigation (per D2-11 slow-test reporter)

**Acceptance criteria.** All identified user-facing flows have at least one happy-path E2E. Flaky tests (> 5 % failure rate over 20 runs) investigated + fixed or explicitly `test.fixme`'d.

**Lock-in.** Already in CI as Playwright job. Expanded tests inherit.

### S7 — Dependency audit + functional tests + file-length gate

**Purpose.** Bundle three small items: `npm audit` / `pip-audit` results, functional-level tests (integration between modules), and a file/function length CI gate.

**Dependency audit (~10 min.)**
1. `npm audit --json > ASSESSMENTS/S7-deps/npm-audit.json`
2. `pip-audit -r app/requirements.txt > ASSESSMENTS/S7-deps/pip-audit.txt` (we have `app/requirements.txt`)
3. Triage: fix all high/critical. Moderate + below with documented assessment.

**Functional tests.** The gap between unit and E2E. Examples: synth_turn.py end-to-end with a fixture transcript but no network (edge-tts mocked), PS hook → main.js IPC contract test. This is usually the thinnest layer; might be empty on our codebase.

**File-length gate (~15 min).**
1. Write `scripts/check-file-length.cjs` — enumerates source files, flags any > 1000 lines (initial threshold relaxed for main.js/renderer.js), fails CI if a file NEWLY crosses the threshold
2. Record current threshold per file in a JSON baseline so we can tighten over time
3. Add to CI as `file-length` job

**Expected findings.**
- npm audit: probably 0–3 low/moderate advisories on dev deps (c8, playwright)
- pip-audit: probably 0 (we pin edge-tts and have minimal deps)
- Functional: 1–3 obvious gaps
- File-length: captures the 1,802 / 1,681 / 2,666 baselines with intent to shrink over v0.4.x

**Acceptance criteria.** Zero high/critical CVEs in deps. File-length baseline recorded. 2–3 functional tests for the most-obvious cross-module invariants.

**Lock-in.** npm audit + pip-audit + file-length all CI-blocking.

## 4. Coordination plan

### Discovery phase (Phase 1) — 5 terminals in parallel

| Terminal | Stream | Language scope | Writes to |
|---|---|---|---|
| T-A | S1 Knip + S5 coverage-baseline | JS + test harness (read-only) | `ASSESSMENTS/S1-knip/` + `ASSESSMENTS/S5-coverage/` |
| T-B | S3 Python/PS lint | .py + .ps1/.psm1 (read-only) | `ASSESSMENTS/S3-py-ps/` |
| T-C | S2 ESLint | JS (read-only for discovery) | `ASSESSMENTS/S2-eslint/` |
| T-D | S4 SonarQube local (once server + scanner are up) | all (read-only) | `ASSESSMENTS/S4-sonar/` |
| T-E | S6 E2E gap + S7 deps + file-length | docs + spec enumeration | `ASSESSMENTS/S6-e2e/` + `ASSESSMENTS/S7-misc/` |

All terminals read-only during discovery → zero conflict risk. `ASSESSMENTS/` folder is new; each stream gets a sub-folder; no overlap.

### Triage phase (Phase 2) — single lane

Ben + me on one terminal. Walk each stream's findings.md. Disposition every entry. Merge into consolidated `ASSESSMENTS/DISPOSITIONS.md`.

### Fix phase (Phase 3) — carefully partitioned

Language-based lanes per existing coord pattern:
- **T-1 (JS/CI):** Knip fixes + ESLint fixes (serial; Knip first). Also SonarCloud-driven JS refactors.
- **T-2 (Python/PS):** ruff fixes + PSScriptAnalyzer fixes. Also SonarCloud-driven Python refactors.
- Tests (S5 gap fill + S6 E2E expansion) go to whichever terminal is lighter-loaded; both edit `scripts/run-tests.cjs` / `tests/` with the island contract (different describe blocks / files).

### Lock-in phase (Phase 4) — sequential

One tool at a time, CI wire-up PR, verified green, merged. Don't batch 5 CI changes into one PR — they fail together and are hard to unstick.

## 6. Open questions before kickoff

1. **Bundle or patch-per-stream?** One v0.4.0 tag is cleaner history; patches are easier to bisect if anything regresses.

4. **Fix-everything exceptions.** Some findings will be genuinely not-worth-fixing (e.g., a complexity warning on a function that's complex for a good reason). Protocol: such findings get `// NOSONAR explanation` comments OR an entry in `ASSESSMENTS/ACCEPTED-SMELLS.md` with the mechanical reason. Policy-level: should Ben pre-approve any "accept"s or can I triage with justification and you review in bulk?
